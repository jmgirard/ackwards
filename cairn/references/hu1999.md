## Why this is a primary source

The source of the **conventional fit-index cutoffs** the package draws as
reference lines on the fit plot and labels in the fit table. `tidy()` and
`autoplot()` cite "Hu & Bentler (1999) thresholds"; these are those thresholds.

## Extracted values

From "CONCLUSION AND RECOMMENDATION" (p. 27), verbatim: a specific cutoff is
hard to designate because it does not work equally well under various
conditions, but —

- "a cutoff value close to .95 for the ML-based TLI, BL89, CFI, RNI, and Gamma
  Hat" — p. 27. → the code's **CFI ≥ .95** and **TLI ≥ .95**.
- "a cutoff value close to .08 for SRMR" — p. 27. → the code's **SRMR ≤ .08**.
- "a cutoff value close to .06 for RMSEA" — p. 27. → the code's **RMSEA ≤ .06**.

All four constants in `.fit_cutoffs()` (`R/tidy.R:283-286`) match these values.

**Two nuances worth carrying** (both p. 27):

1. Hu & Bentler hedge every value with "close to" and stress that no single
   cutoff works universally — so treating `.95`/`.06`/`.08` as hard lines is the
   field's convention, not their prescription. The package already frames them
   as "conventional thresholds" and "rules of thumb, not verdicts", consistent
   with the source.
2. The paper's headline is a **two-index combinational** strategy, and its
   recommended combination pairs TLI/CFI close to `.95` with **SRMR close to
   .09** ("We recommend that practitioners use a cutoff value close to .95 for
   TLI … in combination with a cutoff value close to .09 for SRMR"). The `.08`
   the package uses is the value from the single-index "lower Type II error"
   list on the same page — the more commonly cited figure. Both are p. 27; the
   `.08`/`.09` split is a known source of citation ambiguity.
